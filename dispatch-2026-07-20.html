Daily Technical Snippet

What is approximate nearest-neighbour search (HNSW)?

Once your memories are vectors, retrieval becomes “find the closest points to this query.” Doing that exactly means comparing the query to every stored vector — fine for thousands, hopeless for millions. Approximate nearest-neighbour (ANN) search trades a sliver of accuracy for orders-of-magnitude speed, and HNSW — Hierarchical Navigable Small World — is the index almost every vector database reaches for.

Learn more

HNSW builds a layered graph of your vectors. The top layers are sparse “express lanes” with long-range links; the bottom layer holds everything, densely connected to near neighbours. A search drops in at the top, greedily hops toward the query through the sparse layers to get roughly the right neighbourhood fast, then descends and refines locally — so lookup cost grows logarithmically rather than linearly with corpus size. Two knobs govern the trade-off: M (links per node, more = better recall, more memory) and efSearch (how wide to explore at query time, higher = more accurate, slower). The catch that matters for a memory layer: pure HNSW graphs are awkward to update and filter — deletes leave tombstones, and “only search this user's memories after this date” fights the graph structure. For Kepler, that's the real engineering surface: not “can we do vector search” (a solved commodity) but doing it with per-user isolation, metadata filters and fresh writes without the recall falling over — which is exactly where a URL-first, auditable store earns its keep.

→ Malkov & Yashunin, the original HNSW paper (arXiv)

NVIDIA reframes agent economics around “intelligence per dollar”

At the heart of NVIDIA's Vera Rubin push is a new yardstick: intelligence per dollar, not just cost per token. The argument is that agents demand continuous post-training — they keep meeting new tools and edge cases in production, so training loops back from the field rather than finishing once. Vera Rubin claims to train the largest models on a quarter of the GPUs of the prior Blackwell generation; NVIDIA cites a Nemotron run hitting 71.7% on SWE-bench via RL post-training.

NVIDIA's answer is retrain-cheaper; yours is remember-and-retrieve, and for most teams the second is far cheaper than the first. That's a clean talking point: not everyone can afford a post-training loop on Rubin silicon, but everyone can attach a memory layer that keeps an agent current between model updates. The through-line with today's hero is one story — capability is plateauing and adaptation is the new battleground.

Competitor & Category Watch

Microsoft Research unveils “Memora” to fix agents' memory problem

Microsoft's Memora is a memory architecture that decouples storage from retrieval: each entry pairs a 6–8 word abstraction with the rich value, adds context-aware cue anchors for alternate retrieval paths, and uses a policy-guided retriever that iteratively refines its query rather than one-shot fetching. Microsoft reports up to 98% fewer tokens than full-context inference and 86.3% on LoCoMo / 87.4% on LongMemEval. A pointed line from an analyst: “the deepest flaw in current agent memory is that it mistakes retrieval for memory.” Research project, code on GitHub, not yet productionised.

Why it matters — for you, Jacko

This is the most direct read on your category all week — Microsoft Research is now publicly building the thing Kepler sells, and its framing (“retrieval is not memory”) is almost word-for-word your pitch. Two takeaways. First, the design converges on ideas you should own the language of: separate the durable value from a cheap retrieval key, and refine queries iteratively rather than one-shot — if Kepler doesn't already articulate that, borrow it. Second, and more strategic: a Microsoft research artefact validates the market and signals that the hyperscalers will eventually ship this, so your moat can't be the algorithm — it has to be the things a Microsoft can't easily copy: user-owned, URL-addressable, auditable, portable across every agent. Memora is a lab project today; treat it as a roadmap of where Copilot memory goes.

“Context architecture” is being pitched as the successor to RAG

A running argument across enterprise-AI analysis this quarter: naive RAG — chunk, embed, top-k, stuff” is buckling as agents run longer and demand more relevant context, and is giving way to context architecture: deliberate systems that assemble, order, compress and refresh what an agent sees, blending retrieval with memory, tools and structured state rather than a single similarity lookup. The reframing moves the value from “which vectors matched” to “what context did we actually construct for this step.”

The risk is the term gets co-opted by every incumbent, so stake your specific claim early: URL-first, user-owned, auditable context — the version an enterprise can actually govern.
