Daily Technical Snippet

What is HyDE (Hypothetical Document Embeddings)?

Normal retrieval embeds the user's question and looks for stored text nearby — but a short question and a rich answer often live far apart in vector space. HyDE flips the order: first ask the model to hallucinate a plausible answer to the query, then embed that fake document and search with it. The made-up answer looks far more like the real passage you want than the bare question did, so recall jumps — no labelled training data required.

Learn more

The trick, from Gao et al. (2022), is that you never trust the hypothetical document's content — it can be wrong or invented. You only use its embedding as a better-shaped query vector, letting a dense retriever pull back real, grounded passages that sit near it. It's a query-side transform: cheap, model-agnostic, and stackable on top of whatever index you already run (HNSW, hybrid, rerankers). The costs are a second LLM call per query and sensitivity to a badly-drifting hallucination, which is why production systems pair it with reranking and keep the raw query in the mix. For Kepler the lesson is where the leverage sits: once memories are stored as clean, URL-addressable, auditable objects, most retrieval-quality gains come from shaping the query against them — HyDE, expansion, rerank — not from re-embedding the store. A stable, portable memory substrate is exactly what lets those query-side tricks compound.

→ Gao et al., “Precise Zero-Shot Dense Retrieval without Relevance Labels” (arXiv)
