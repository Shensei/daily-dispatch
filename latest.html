Daily Technical Snippet

What is "sleep-time compute" (offline memory consolidation)?

Agents waste tokens re-deriving the same context every turn. Sleep-time compute flips that: while the agent is idle, a background process pre-chews raw memory — summarising, linking entities, resolving contradictions — so that at query time the answer is already half-built. It trades cheap offline cycles for fast, cheap online ones; Letta's research showed it can cut per-query cost several-fold at equal accuracy, and it's why Oracle and others now ship async "consolidation".

Learn more

Test-time compute made models "think harder" during a query. Sleep-time compute asks a different question: what if the model thinks ahead of the query? Between sessions, a background agent reprocesses the raw log — clustering related facts, writing durable summaries, pruning the dead, and reconciling anything that changed — so the live context window is small, clean and pre-indexed when the user finally asks. In the Letta study this cut the tokens needed at inference by roughly 5× for the same answer quality, and amortised across many queries it's near-free. This is the natural spine for Kepler: a URL-first memory can do its consolidation offline and publish the result, so an agent fetches a compact, already-reconciled context by reference — provably, and without paying to rebuild it every turn.
