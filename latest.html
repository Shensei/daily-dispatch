Daily Technical Snippet

What are Matryoshka embeddings?

A normal embedding is all-or-nothing: a 1,536-dimension vector only makes sense at its full length. Matryoshka embeddings are trained so the most important information is packed into the first dimensions — like nested Russian dolls — so you can truncate a vector to 256 or 512 dims and it still works, just slightly less precisely.

Learn more

Matryoshka Representation Learning (MRL) adds a loss term at multiple nested prefixes (say 64/128/256/512/1024) during training, so a single model produces one vector that degrades gracefully as you chop it shorter. The payoff is adaptive retrieval: store the full-length vector once, then do a fast, cheap first pass over truncated 128-dim versions to shortlist candidates, and only re-rank the survivors at full precision. You cut memory and index cost by 4–8× with almost no accuracy loss — which is exactly the kind of retrieval-cost lever a URL-first memory layer like Kepler lives or dies on when it has to recall across a user's whole history without blowing the token and latency budget. OpenAI's text-embedding-3 and Nomic's models already ship with this "shorten the dimensions" knob built in.

MCP's biggest-ever rewrite lands as a release candidate — final spec ships 28 July

The Model Context Protocol's 2026-07-28 revision goes stateless at the protocol layer: the initialize/session handshake is gone, so any request can hit any server instance behind a plain load balancer. It also makes Extensions first-class (MCP Apps for server-rendered UIs, a redesigned Tasks extension), hardens OAuth/OIDC authorization, and adds cacheable tools/list with a formal deprecation policy. Breaking changes; final lands 28 Jul.
