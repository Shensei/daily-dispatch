Daily Technical Snippet

What is prompt caching (KV-cache reuse)?

When an LLM reads your prompt, it computes a big intermediate structure called the KV cache — one entry per token, per layer. Prompt caching saves that work for a stable prefix (your system prompt, a document, a memory bundle) so the next call skips re-processing it and starts from where it left off. Same output, a fraction of the cost and latency.

Learn more

Attention is quadratic on the way in: processing 50k tokens of context costs real compute every single call, even when 49k of it never changed. Providers now let you mark a prefix as cacheable — Anthropic with explicit cache_control breakpoints, OpenAI and Google with automatic prefix caching — and cached input tokens bill at roughly 10% of normal input price with large time-to-first-token wins. The catch is that caching is prefix-exact: the cached span must be byte-identical and at the front, so a single changed character early in the prompt invalidates everything after it, and caches expire in minutes. The design lesson for a memory layer like Kepler is ordering: put the stable stuff first (system prompt, then long-lived retrieved context), and the volatile stuff last (the user's turn). Get that ordering wrong and you pay full freight on every call; get it right and a large, always-on context bundle stops being a cost problem — which meaningfully changes the "is rich memory too expensive?" objection.

Top Story · AI & Agents

Google and a dozen partners ship "Agentic Resource Discovery" — the layer that tells agents which tools exist

MCP defines how an agent calls a tool. ARD tackles the step before: how it finds one. Organisations publish a machine-readable ai-catalog.json describing their tools, APIs, skills and agent endpoints; registries aggregate those catalogs so agents can search by task intent instead of hardcoded integrations — with domain-based ownership and verification so a resource can be trusted before connection. Contributors include Microsoft, GitHub, Hugging Face, Cisco, Databricks, NVIDIA, Salesforce, ServiceNow and Snowflake. GitHub's Agent Finder and Hugging Face's Discover Tool already ship on it.

Why it matters — for you, Jacko

This is a distribution question disguised as a spec. If agents start discovering capabilities through catalogs and registries rather than a user manually installing a connector, then "is Kepler in the registry, and does it describe itself well?" becomes a growth channel — publishing an ai-catalog.json is arguably the cheapest GTM work on your list this quarter, and it's explicitly complementary to MCP, so it doesn't cost you a rewrite. The trust half matters too: ARD's domain verification is the kind of thing enterprise buyers ask about, and it rewards being the memory layer with provable provenance. For the Alinta seat, this is the shape of the governed-discovery problem any large enterprise hits the moment it has more than a handful of internal agents.

AI & Agents

Stripe's benchmark: agents can write the integration, but can't tell whether it works

Stripe open-sourced a suite of 11 reproducible environments — real codebases, databases and test API keys — to see whether agents can build production-style Stripe integrations end to end, running through a Goose + MCP harness with terminal, browser and docs access. Claude Opus 4.5 averaged 92% on full-stack API integration tasks; GPT 5.2 hit 73% on structured tasks. The failures cluster in validation: agents see an expected HTTP 400 from invalid input and conclude success, or lose browser state mid-checkout and quit rather than recover.

Why it matters — for you, Jacko

"Generation is solved, validation isn't" is the most useful sentence in agent-land right now, and it's a memory problem in disguise: agents fail here because they lose track of state across a long horizon and can't recall what they already tried or what "correct" looked like. That's a sharper, more technical wedge for Kepler than any productivity framing — durable, retrievable context is what lets an agent check its own work. Worth stealing the framing for content: a benchmark showing best runs sustained ~63 turns before correctness degraded is exactly the evidence a memory-layer pitch wants.

MCP's biggest-ever rewrite is locked — final spec ships 28 July

The 2026-07-28 revision makes MCP stateless at the protocol layer: the initialize/session handshake is gone, so any request can land on any server instance behind a plain load balancer. Extensions become first-class (MCP Apps for server-rendered UIs, a redesigned Tasks extension), authorization aligns closer to OAuth/OIDC, tools/list results gain ttlMs caching, and Roots/Sampling/Logging are deprecated under a new 12-month lifecycle policy. Breaking changes; the RC has been locked since 21 May.

Being conformant on day one, while most connectors are still mid-migration, is a cheap credibility win with the developer audience that decides which memory tool gets installed.

MCP's enterprise auth extension goes stable — one login for every connected server

The Model Context Protocol team promoted its Enterprise-Managed Authorization extension to stable, moving the connect decision into the company's identity provider so users sign in once and inherit access to approved servers instead of clicking through per-server consent prompts. It uses an Identity Assertion JWT grant (ID-JAG); Okta is the first IdP. Anthropic (Claude, Claude Code, Cowork), Microsoft and VS Code have implemented it, with Asana, Atlassian, Canva, Figma, Granola, Linear and Supabase supporting it server-side and Slack in progress.

Why it matters — for you, Jacko

Repeated consent prompts are the single biggest drop-off in enterprise MCP adoption, so this removes a real activation tax on Kepler — but only for servers that implement it, and the named early adopters are all the tools an IT admin already trusts. Being on that list is the difference between "approved once for everyone" and "each employee has to set it up," which is effectively a distribution decision. Worth reading the caveat too: EMA governs whether you can connect, not what happens after the token is issued, so runtime controls over what a memory layer can read and write are still yours to own — and to sell.

Energy & Utilities · Alinta Watch

Alinta's "AISE" is in production — confidence-scored answers for the executive team

At Databricks' Data + AI Summit, Alinta's GM of data & AI Brad Walker detailed the Alinta Intelligent Strategic Engine — built in about four weeks and ~70 person-days out of a collaboration between Alinta's CEO and CIO. A supervising agent routes each executive question across 500 structured metrics and 9,000+ documents (board papers, strategy decks, market research). Every answer carries a confidence score — "you can make a decision on this" versus "directional, good for your thinking" — plus a link to the exact metric or document behind it. Now in production, with per-business-unit agents being built out.
