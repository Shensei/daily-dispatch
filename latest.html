Daily Technical Snippet

What is Agentic RAG?

Classic RAG runs one fixed step: embed the question, pull the top-k chunks, stuff them in the prompt. Agentic RAG makes retrieval a tool the model decides when and how to use — it can reformulate the query, search several sources, read a result and search again, or skip retrieval entirely when it already knows. Retrieval stops being a pipeline stage and becomes a loop the agent controls.

Learn more

The shift is from retrieve-then-generate to reason–retrieve–reason: the agent plans, calls a search or tool, judges whether the evidence is enough, and iterates — optionally routing across many indexes or handing sub-queries to other agents. It trades latency and cost (several LLM-plus-retrieval calls) for much better recall on multi-hop and ambiguous questions, and it degrades gracefully because the agent can notice a bad result and try again. The catch: every extra hop needs somewhere reliable to look — an agent that re-queries five times is only as good as the memory it queries. For Kepler that's the whole point. If each agentic hop reads from a clean, URL-addressable, auditable memory instead of a black-box vector blob, the loop stays fast, cheap to trust, and inspectable when it goes wrong — which is exactly when someone asks why the agent did what it did.

→ Singh et al., “Agentic Retrieval-Augmented Generation: A Survey on Agentic RAG” (arXiv)
