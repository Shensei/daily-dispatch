Daily Technical Snippet

What is reranking?

Reranking is the second pass in a good retrieval system. You first pull, say, the top 50 candidate chunks cheaply with vector search, then a heavier reranker model re-scores those 50 by how well each actually answers the query — and you keep only the best 5 to feed the model. It's the step that turns "roughly related" into "actually right," and for a URL-first memory product it's often the difference between surfacing the paragraph that matters and burying it at rank 40.

Learn more

Embeddings are fast because query and document are encoded separately — great for scanning millions of chunks, but the query never "sees" the document. A cross-encoder reranker fixes that: it feeds the query and each candidate together through a transformer, so it can judge fine-grained relevance (negation, specificity, intent) that a cosine-similarity score misses. The trade-off is cost — you can't cross-encode your whole corpus, so the pattern is always retrieve wide, rerank narrow. In practice a reranker is the cheapest single upgrade to RAG quality: same index, same embeddings, but the right context lands in the prompt far more often. For Kepler, retrieval + reranking quality basically is the product — it decides whether the correct URL surfaces at the moment it's needed.
